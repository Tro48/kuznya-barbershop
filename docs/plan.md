# Проект №1 — «Кузня», барбершоп

> Читать вместе с [`common.md`](./common.md): стек, структура, защита форм,
> доступность и правила деплоя описаны там и здесь не повторяются.

## Анализ

### Что на самом деле продаёт этот лендинг

Не барбершоп. Он продаёт **уверенность, что не испортят**. Мужчина, выбирающий
новое место для стрижки, боится ровно одного: выйти с головой, которую придётся две
недели прятать. Всё остальное — цена, интерьер, кофе — вторично.

Из этого следует иерархия доверия на странице, и она отличается от порядка блоков
в исходном ТЗ:

1. **Работы мастеров** — единственное настоящее доказательство. Должны быть выше,
   чем «Почему мы».
2. **Мастера как люди** — с именами и лицами. Стрижёт человек, а не «барбершоп».
3. **Отзывы** — подтверждение чужим опытом.
4. **Преимущества с иконками** — самый слабый блок, его никто не читает.

### Что меняю в ТЗ и почему

| Было в ТЗ | Стало | Причина |
|-----------|-------|---------|
| «Почему мы» — 3–4 преимущества с иконками | Блок «Мастера» с фото и специализацией | Иконка ножниц с подписью «качество» не убеждает никого. Лицо мастера с подписью «12 лет, специализация — фейд» убеждает. |
| Галерея — просто фото | Галерея с фильтром по мастеру | Даёт связку «работа → мастер → запись именно к нему». Фильтр — 20 строк кода, конверсионный эффект ощутимый. |
| Форма записи — имя, телефон, услуга, комментарий | То же + выбор мастера (опционально) и предпочтительное время | Два поля, которые реально нужны администратору. Без них он всё равно перезвонит и спросит. |
| Карта + режим работы | То же, но карта грузится по клику | Iframe Яндекс.Карт весит больше остального лендинга (см. `common.md`). |

Блок преимуществ не удаляю совсем — сворачиваю в тонкую строку из трёх фактов
(«с 2016 года», «без записи по будням до 12:00», «детская стрижка — с 3 лет») под
героем. Факты вместо прилагательных.

### Техническая ценность для портфолио

Заявка уходит в Telegram и приходит администратору за секунду. Это то, что отличает
работающий лендинг от вёрстки. Показываем на собеседовании:

- Route Handler с серверной валидацией той же Zod-схемой, что и клиент.
- Токен бота недоступен из браузера (`server-only`, проверяемо через поиск по бандлу).
- Обработка отказа Telegram API: пользователь видит «позвоните нам», а не «Error 500».
- Идемпотентность: повторный сабмит той же формы не создаёт вторую заявку.

Последний пункт — то, о чём забывают. Пользователь жмёт кнопку дважды, потому что
не понял, что первый клик сработал. Решается блокировкой кнопки на время запроса и
дедупликацией по хешу `(phone, service, минута)` на сервере.

### Риски

- **Telegram API недоступен из региона.** Vercel-функции ходят из дата-центров, где
  доступ обычно есть, но гарантий нет. Резервный путь: при ошибке отправки пишем
  заявку в Supabase и показываем успех — заявка не теряется. Это разумно и на реальном
  проекте.
- **Фото-контент.** Лендинг барбершопа без хороших фото мёртв. Заложить час на подбор
  8–10 снимков одной цветовой температуры — разнобой по свету убьёт галерею.
- **Дизайн-клише.** «Тёмный барбершоп с золотом» — самый заезженный шаблон в нише.
  Отстройка ниже, в разделе про дизайн.

## Название

**Кузня**

Домен: `kuznya.<ваш-домен>` или `kuznya-barbershop.vercel.app`.

Почему работает: мужское ремесло, огонь и сталь, работа руками — то же поле смыслов,
что бритва и ножницы, но без прямого называния. Слово короткое, произносится с
ударением на первый слог, легко запоминается на слух — это важно, потому что
барбершоп советуют голосом. Даёт визуальный язык бесплатно: сталь, окалина, латунь,
искра, наковальня.

Слоган: **«Работа по стали и волосу»** — сдержанно, с иронией, без «мы лучшие в городе».

Запасные варианты, если «Кузня» покажется занятой:

- **Обух** — тыльная сторона топора. Жёстко, коротко, свободно.
- **Верста** — старая мера длины, ассоциация с «верстак». Мягче, интеллигентнее.
- **Наждак** — фактурно, но рискует читаться как «грубо».

Имена придуманы для демо и **не проверялись** по реестру товарных знаков.

## Дизайн

### Отстройка от шаблона

Стандартный барбершоп-лендинг: чёрный фон, золотые вензеля, шрифт с засечками,
фотографии в сепии, иконки ножниц и усов. Делаем иначе — **не «винтажный
джентльмен», а «мастерская»**:

- Не золото, а **латунь** — приглушённая, почти охра. Золото блестит, латунь работает.
- Не вензеля, а **сетка и линии** как на чертеже. Тонкие рамки 1px.
- Не сепия, а **холодный контрастный свет** на фото. Мастерская, а не парикмахерская 1920-х.
- Не курсив с завитками, а **узкий гротеск**, набранный крупно и капслоком. Как
  трафаретная надпись на ящике с инструментом.
- Никаких иконок ножниц и усов. Совсем.

### Палитра

Тёмная тема — единственная. Светлого варианта нет: у мастерской нет светлого режима.

```css
@theme {
  --color-coal:    #0F0E0D; /* фон */
  --color-anvil:   #1A1816; /* поверхность карточек */
  --color-scale:   #2E2A26; /* границы, разделители */
  --color-chalk:   #F2EDE6; /* основной текст */
  --color-ash:     #A39B90; /* вторичный текст */
  --color-brass:   #C8963E; /* акцент */
  --color-spark:   #DDAA4C; /* акцент: ховер */
  --color-danger:  #E5654B; /* ошибки формы */
}
```

Контраст проверен: `chalk` на `coal` — 15.8:1. `brass` на `coal` — 6.7:1 (проходит
для текста). `ash` на `coal` — 7.1:1. Кнопка — фон `brass`, текст `coal`: 6.7:1.
Все значения выше порога WCAG AA.

Латунь — **только для действия**. Если акцентным цветом покрашены заголовок, иконка
и кнопка, кнопку перестают замечать. На первом экране латунь встречается ровно один
раз: на кнопке «Записаться».

### Типографика

- Заголовки: **Oswald** 600/700, `letter-spacing: 0.02em`, капслок для h1/h2.
  Узкий гротеск с кириллицей, бесплатный, ставится через `next/font/google`.
- Текст: **Golos Text** 400/500. Спокойный, отлично читается на тёмном.
- Цифры цен: Golos Text 500 с `font-variant-numeric: tabular-nums` — колонка цен
  должна выравниваться по разрядам.

Масштаб (mobile → desktop, через `clamp`):

```css
--text-hero:  clamp(2.75rem, 9vw, 6rem);    /* h1, капслок, line-height: 0.95 */
--text-h2:    clamp(1.75rem, 4vw, 3rem);
--text-h3:    1.25rem;
--text-body:  1.0625rem;  /* 17px — на тёмном фоне 16px читается мельче */
--text-small: 0.875rem;
```

`line-height: 0.95` на герое — то, что делает заголовок «плакатным». На body всегда
`1.6`, не меньше: светлый текст на тёмном фоне визуально «слипается».

### Сетка и пространство

Контейнер `max-width: 1200px`, боковые поля `24px` на мобильном, `48px` на десктопе.
Вертикальный ритм секций: `clamp(72px, 10vw, 140px)` сверху и снизу.

Скругления: `4px` максимум. Кузня не про мягкость. Карточки услуг — прямоугольные,
с рамкой `1px solid var(--color-scale)`.

Тени не используем вообще. На угольном фоне тень не видна, глубину даёт граница и
чуть более светлая поверхность (`anvil` против `coal`).

### Анимации

Сдержанные, «тяжёлые». Кузнечный молот не порхает.

- Появление секций: `opacity 0 → 1`, `y: 24px → 0`, `duration: 0.5`, `ease: [0.16, 1, 0.3, 1]`.
- Стагger по карточкам услуг: `0.06s` между элементами. Больше — начинает раздражать.
- Ховер карточки: граница `scale` → `brass`, `transition: 200ms`. Без `transform: scale()`.
- Единственная «дорогая» анимация — счётчик в строке фактов под героем (число лет
  работы отсчитывается вверх при появлении). Одна такая деталь на лендинг, не три.

Всё под `prefers-reduced-motion` (см. `common.md`).

## Структура страницы

Порядок отличается от исходного ТЗ по причинам из раздела «Анализ».

1. **Header** — липкий, прозрачный на герое, с фоном `coal/90 + backdrop-blur` после
   скролла на 80px. Логотип, телефон (кликабельный `tel:`), кнопка «Записаться».
2. **Hero** — экран целиком (`min-height: 100svh`, не `100vh`: на мобильных `vh`
   считает адресную строку и герой уезжает). Фото мастерской, поверх — заголовок
   капслоком, оффер в строку, кнопка + телефон.
3. **Строка фактов** — три факта в ряд, тонкие разделители. Заменяет «Почему мы».
4. **Услуги** — 6 карточек: стрижка, борода, стрижка + борода, бритьё опасной бритвой,
   камуфляж седины, детская. Цена «от», время выполнения, кнопка «Записаться»
   предзаполняет услугу в форме.
5. **Работы** — галерея 8–10 фото с фильтром по мастеру. Клик — лайтбокс.
6. **Мастера** — 3 карточки: фото, имя, стаж, специализация, кнопка «Записаться к нему».
7. **Отзывы** — 4 карточки, горизонтальный скролл на мобильном (`scroll-snap`), сетка на десктопе.
8. **Карта и режим работы** — заглушка карты, адрес, часы, ориентир («вход со двора»).
9. **Форма записи** — главный экран страницы. Разбор ниже.
10. **Footer** — контакты, соцсети, ссылка на `/privacy`, строка про демо-проект.

Мобильное меню — не гамбургер с оверлеем. На лендинге из девяти секций навигация не
нужна: в шапке достаточно телефона и кнопки записи. Меньше кода, лучше конверсия.

## Форма записи

Поля: имя, телефон, услуга (select, предзаполняется из карточки), мастер (select,
«не важно» по умолчанию), удобное время (chips: «утро / день / вечер»), комментарий,
чекбокс согласия.

Ключевые решения:

- **Маска телефона.** Пишем свою на `onChange` вместо библиотеки: нормализуем ввод
  в `+7XXXXXXXXXX`, показываем `+7 (999) 123-45-67`. Библиотеки масок ломают вставку
  из буфера и `autofill`. Свой обработчик из 15 строк ведёт себя предсказуемо.
- **`inputMode="tel"` и `autoComplete="tel"`** — на мобильном открывается цифровая
  клавиатура, автозаполнение работает. Мелочь, которую замечают.
- **Предзаполнение услуги** через Zustand не нужно: достаточно поднять состояние в
  `page.tsx` или прокинуть через `useState` + скролл к форме. Zustand в этом проекте
  не используем сознательно — он появится в проекте 2, где оправдан.
- **Состояния кнопки**: `idle → pending (спиннер, disabled) → success (галка) → error`.
  Возврат из `success` в `idle` не делаем: показываем экран благодарности вместо формы.
- **Экран успеха** сообщает, что будет дальше: «Перезвоним в течение 15 минут. Если
  сейчас ночь — утром». Это снимает тревогу и снижает число повторных заявок.

## Интеграция с Telegram

```
Клиент → POST /api/lead → Zod-валидация → honeypot + time-trap → rate limit
       → sendMessage в Telegram → 200 OK
                              ↘ ошибка → запись в Supabase → 200 OK (заявка не потеряна)
                                       ↘ обе упали → 503 + текст «позвоните нам»
```

Настройка бота:

1. `@BotFather` → `/newbot` → получаем `TELEGRAM_BOT_TOKEN`.
2. Создаём группу «Кузня — заявки», добавляем бота, даём права на отправку.
3. `TELEGRAM_CHAT_ID` группы: отправить сообщение в группу и прочитать
   `https://api.telegram.org/bot<TOKEN>/getUpdates` — `chat.id` будет отрицательным.

Route Handler:

```ts
// app/api/lead/route.ts
import "server-only";
import { NextResponse } from "next/server";
import { leadSchema } from "@/lib/schemas/lead";
import { env } from "@/lib/env";

export async function POST(req: Request) {
  const parsed = leadSchema.safeParse(await req.json());
  if (!parsed.success) {
    return NextResponse.json({ error: "invalid" }, { status: 400 });
  }
  const { website, ...lead } = parsed.data;
  if (website) return NextResponse.json({ ok: true }); // бот: молча "успех"

  const text = [
    "🔥 *Новая заявка — Кузня*",
    `Имя: ${escapeMd(lead.name)}`,
    `Телефон: \`${lead.phone}\``,
    `Услуга: ${escapeMd(lead.service)}`,
    lead.master ? `Мастер: ${escapeMd(lead.master)}` : null,
    lead.comment ? `Комментарий: ${escapeMd(lead.comment)}` : null,
  ].filter(Boolean).join("\n");

  const res = await fetch(
    `https://api.telegram.org/bot${env.TELEGRAM_BOT_TOKEN}/sendMessage`,
    {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        chat_id: env.TELEGRAM_CHAT_ID,
        text,
        parse_mode: "MarkdownV2",
      }),
      signal: AbortSignal.timeout(5000),
    },
  );

  if (!res.ok) {
    await saveToSupabase(lead); // резервный путь
  }
  return NextResponse.json({ ok: true });
}
```

Два подводных камня, оба стоят потерянных вечеров:

- **`parse_mode: MarkdownV2` требует экранирования** символов `_ * [ ] ( ) ~ \` > # + - = | { } . !` в
  *любом* тексте, включая пользовательский. Имя «Пётр-Иван» уронит запрос с 400.
  Отсюда `escapeMd()`. Альтернатива — не использовать `parse_mode` вообще; для
  заявки форматирование не критично, и это честно проще.
- **Таймаут.** Без `AbortSignal.timeout()` зависший запрос к Telegram держит
  функцию до лимита Vercel, а пользователь смотрит на спиннер. 5 секунд — потолок.

## Данные и контент

```ts
// content/services.ts
export type Service = {
  slug: string;
  title: string;
  priceFrom: number;   // в рублях, целое
  durationMin: number;
  description: string;
};
```

Цены пишем как числа и форматируем через `Intl.NumberFormat("ru-RU")` — не строкой
`"от 1 500 ₽"`. Форматирование в одном месте, сортировка возможна, опечатки в
пробелах исключены.

Аналогично `content/masters.ts`, `content/reviews.ts`, `content/faq.ts`.

## План реализации

Оценки — чистое время, без отвлечений.

### Шаг 1. Подготовка — 2 ч
- Репозиторий из шаблона, `pnpm install`, CI зелёный.
- `lib/env.ts` со схемой переменных.
- Токены темы в `globals.css`, шрифты через `next/font/google`.
- Проверка: `pnpm build` проходит на пустой странице.

### Шаг 2. Контент — 2 ч
- Тексты всех секций в `content/`. Все шесть услуг с ценами и временем.
- Подбор 10 фото работ + 3 фото мастеров + 1 фото героя. Одна цветовая температура.
- Прогон фото через `squoosh` в WebP, максимум 1600px по длинной стороне.

### Шаг 3. Примитивы UI — 2 ч
- `Button` (варианты `primary` / `ghost`, состояния `pending` / `disabled`).
- `Input`, `Select`, `Textarea`, `Checkbox` — с `aria-invalid` и `aria-describedby`.
- `Container`, `Section` (отвечает за вертикальный ритм).
- Критерий готовности: вкладкой проходятся все элементы, фокус виден.

### Шаг 4. Секции — 6 ч
Сверху вниз, каждая секция — отдельный коммит. Порядок: Header → Hero → Факты →
Услуги → Работы → Мастера → Отзывы → Карта → Footer.

Галерея с фильтром — самая долгая (1.5 ч): состояние фильтра, `AnimatePresence` с
`layout` для перестановки карточек, лайтбокс с закрытием по `Esc` и ловушкой фокуса.

### Шаг 5. Форма и Telegram — 4 ч
- Zod-схема + юнит-тесты на неё (валидные и невалидные телефоны, honeypot, согласие).
- React Hook Form, маска телефона, предзаполнение услуги.
- Route Handler, экранирование MarkdownV2, таймаут, резервная запись в Supabase.
- Rate limit через Upstash.
- Проверка: заявка приходит в Telegram; при подменённом токене — падает в Supabase
  и пользователь всё равно видит успех.

### Шаг 6. Полировка — 3 ч
- Анимации появления, счётчик в фактах, `prefers-reduced-motion`.
- Карта по клику вместо iframe сразу.
- Lighthouse на продовом билде, добивание до целей из `common.md`.
- Клавиатурная навигация целиком: от логотипа до кнопки отправки.

### Шаг 7. Метаданные и деплой — 2 ч
- `metadata`, OG-картинка через `ImageResponse`, JSON-LD `LocalBusiness` с
  `openingHoursSpecification` и `geo`.
- `sitemap.ts`, `robots.ts`, страница `/privacy`.
- Яндекс.Метрика, проверка цели `form_submit_success` на проде.
- Playwright smoke: заполнить форму → замокать `/api/lead` → увидеть экран успеха.
- README с обоснованием решений.

**Итого: 21 ч.** Верхняя граница оценки — реалистичнее нижней, если фото ищутся долго.

## Критерии готовности

- [ ] Заявка из формы приходит в Telegram за < 3 секунд.
- [ ] Токена бота нет в клиентском бандле (`grep` по `.next/static` пуст).
- [ ] Форма не отправляется без согласия на обработку данных.
- [ ] Повторный клик по кнопке не создаёт вторую заявку.
- [ ] При недоступном Telegram заявка сохраняется в Supabase, пользователь видит успех.
- [ ] Lighthouse mobile: Performance ≥ 90, Accessibility = 100.
- [ ] Вся страница проходится с клавиатуры, фокус виден везде.
